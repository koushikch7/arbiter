Quick Start

Arbiter exposes an OpenAI-compatible API. Any SDK or tool that works with the OpenAI API works with Arbiter — just change the base URL.

Base URL: https://arbiter.chkoushik.com
Endpoint: /v1/chat/completions

Python (OpenAI SDK)

pythonfrom openai import OpenAI client = OpenAI( base_url="https://arbiter.chkoushik.com/v1", api_key="your-gateway-token", # from Settings → Gateway Tokens ) response = client.chat.completions.create( model="auto", # let Arbiter choose the best model messages=[{"role": "user", "content": "Hello!"}], ) print(response.choices[0].message.content)

cURL

bashcurl -X POST https://arbiter.chkoushik.com/v1/chat/completions \ -H "Authorization: Bearer your-gateway-token" \ -H "Content-Type: application/json" \ -d '{ "model": "auto", "messages": [{"role": "user", "content": "Hello!"}] }'

Model Selection

Model ValueBehavior

"gemini-2.5-flash"Use a specific model — Arbiter routes to the correct provider

⚠️ Free Tier Mode: By default, Arbiter routes through free API providers (Gemini, Groq, NVIDIA, etc.). These have rate limits and may not be suitable for production workloads. Configure paid API keys in Settings for production stability.

Authentication

All /v1/* endpoints require authentication via Bearer token.

Gateway API Token

httpAuthorization: Bearer arbiter-sk-xxxxxxxxxxxxxx

Create tokens in Settings → Gateway Tokens. Each token has:

  • Unique name for identification
  • Optional expiry date
  • Enable/disable toggle
  • Per-token analytics (requests, models used, providers hit)
  • Routing policy — restrict which models this token can use

Routing Policies (per token)

PolicyBehavior
autoArbiter chooses freely from all available models (default)
restrictedOnly models in the token's allowed_models list are used. Order = priority.
preferredPreferred models tried first, falls back to full auto if exhausted

Streaming (SSE)

Set "stream": true to receive the response as a text/event-stream (Server-Sent Events) sequence. The format is identical to the OpenAI streaming API — every OpenAI SDK, LiteLLM, LangChain, and direct EventSource works out of the box. All 14 providers are supported natively.

bashcurl -sS -N \ -H "Authorization: Bearer <gateway-token>" \ -H "Content-Type: application/json" \ -d '{"model":"auto","messages":[{"role":"user","content":"Hello!"}],"stream":true}' \ https://arbiter.chkoushik.com/v1/chat/completions

Each data: line carries a chat.completion.chunk object:

sse: arbiter-model-used: gemini/gemini-2.5-flash-lite data: {"id":"chatcmpl-abc","object":"chat.completion.chunk","choices":[{"delta":{"role":"assistant"},"finish_reason":null}]} data: {"id":"chatcmpl-abc","object":"chat.completion.chunk","choices":[{"delta":{"content":"Hello"},"finish_reason":null}]} data: {"id":"chatcmpl-abc","object":"chat.completion.chunk","choices":[{"delta":{},"finish_reason":"stop"}],"usage":{"prompt_tokens":3,"completion_tokens":1,"total_tokens":4}} data: [DONE]
SSE eventMeaning
: arbiter-model-used: <provider/model>SSE comment (ignored by OpenAI SDKs) — identifies which provider fulfilled the request. cache/<model> = served from Redis cache.
: thinking / evaluating / generating / almost thereKeepalive heartbeat emitted every ~5 s while the upstream is thinking (prevents nginx/Cloudflare idle-timeout). Invisible to OpenAI SDKs.
data: {"choices":[{"delta":{"content":"..."}}]}Text delta — append to your buffer.
data: {"choices":[{"delta":{},"finish_reason":"stop"}],"usage":{...}}Final chunk — includes real token usage for all providers that support it.
data: [DONE]Stream end marker.
data: {"error":{"message":"...","type":"...","code":502}}Mid-stream error (rare — only when upstream fails after the first chunk was already sent). Stream ends immediately after.

Caching with streaming: Cache reads replay as an instant stream (milliseconds). Cache writes happen after [DONE] — a non-stream call that populates the cache will make the next stream call instant, and vice versa. Only requests with temperature ≤ 0.3 are cached.

Fallback during streaming: Pre-first-chunk failures fall back transparently to the next provider. Post-first-chunk failures emit a data: {"error":{...}} event and end the stream (bytes already sent, cannot redirect).

Response Headers

HeaderDescription
X-Arbiter-Model-UsedThe actual provider/model that fulfilled the request (e.g. nvidia/nemotron-3-super-120b-a12b)

Routing Algorithm

Arbiter uses a multi-level intelligent routing system:

1. Cache Check

For requests with temperature ≤ 0.3, check Redis cache first. Instant response on hit.

2. Candidate Chain

Build an ordered list of (provider, model) pairs based on:

  • Explicit model name → route to owning provider
  • Token count → prefer providers with sufficient context window
  • Capability tags → code, reasoning, creative → different models
  • Gateway routing policy → filter/reorder based on token config

3. Gateway Policy Filter

If the calling token has routing_policy: "restricted", only allowed models survive. If "preferred", preferred models are moved to the front of the chain.

4. Key Rotation (per model)

For each candidate, rotate through all API keys using weighted scoring:

formulascore = (rpm_available × 0.25) + (tpm_available × 0.15) + (daily_available × 0.40) + (health × 0.20)

5. Fallback

On rate limit → try next key. On provider error → try next model. All exhausted → 503.

Provider Priority (default)

textnvidia → gemini → groq → cerebras → zai → cloudflare → openrouter → cohere → huggingface → pollinations → ollama → routeway

Adaptive Provider Ordering (v1.18.0+)

At runtime, providers with a lifetime error rate ≥ 20% (minimum 100 requests sampled) are automatically demoted to the tail of the candidate chain. The demotion is cached for 60 seconds and re-evaluated continuously. Override the static priority in Settings → Routing.

TPM-aware Key Rotation (v1.18.0+)

The key-picker now accounts for each key's tokens-per-minute budget. When the router estimates that the current request exceeds a key's remaining TPM headroom, that key is deprioritised regardless of its RPM score. If all keys are near their RPM cap and the minute boundary is within 10 seconds, the gateway waits for reset instead of falling through to the next provider.

Gateway Token Management

Gateway tokens control access to Arbiter's completions API. Each token can have its own routing policy.

Create a Token with Model Restrictions

bash# Create a token restricted to specific models curl -X POST https://arbiter.chkoushik.com/api/gateway/tokens \ -H "Cookie: arbiter_session=..." \ -H "Content-Type: application/json" \ -d '{ "name": "OpenClaw Production", "routing_policy": "restricted", "allowed_models": [ "nvidia/nemotron-3-super-120b-a12b", "mistralai/mistral-medium-3.5-128b", "meta/llama-3.3-70b-instruct" ] }'

Update Routing Policy

bash# Switch to preferred mode (try these first, fall back to auto) curl -X PATCH https://arbiter.chkoushik.com/api/gateway/tokens/gwtk_abc123 \ -H "Cookie: arbiter_session=..." \ -H "Content-Type: application/json" \ -d '{ "routing_policy": "preferred", "allowed_models": ["nvidia/nemotron-3-super-120b-a12b"] }'

Block Specific Models

json{ "blocked_models": ["gemini-2.5-pro", "gpt-4"] }

Blocked models are never used regardless of routing policy. Useful for preventing accidental use of expensive paid models.

Activity Audit Security

Every activity record carries an HMAC-SHA256 tag keyed by SESSION_SECRET_KEY. Bearer tokens, API keys, and password-like values are auto-redacted with a head4…tail4 + sha256[:12] fingerprint before write. You can verify the HMAC in Python:

pythonimport hmac, hashlib, json SECRET = b"your-SESSION_SECRET_KEY" with open("data/logs/activity/2026-05-12.jsonl") as f: for line in f: rec = json.loads(line) tag = rec.pop("hmac", "") expected = hmac.new(SECRET, json.dumps(rec, sort_keys=True).encode(), hashlib.sha256).hexdigest() print("OK" if hmac.compare_digest(tag, expected) else "TAMPERED", rec["action"])

Per-Token Rate Limiting (v1.18.0+)

Every /v1/* request is checked against a sliding-minute-window rate limit after Bearer validation.

SettingDefaultHow to change
Global default100 req/minGATEWAY_TOKEN_RATE_LIMIT_PER_MIN in .env
Per-token override—Set request_limit_per_minute on the token via PATCH /api/gateway/tokens/{id}
Disable for a token—Set request_limit_per_minute: 0

When the limit is exceeded the gateway returns 429 Too Many Requests with Retry-After, X-RateLimit-Limit, and X-RateLimit-Remaining headers.

Weekly AI Analysis Email

Every Monday the daily 22:00 IST report includes a weekly section with 7-day API totals, p50/p95 latency, calls-by-provider, errors-by-category, and an AI-generated SRE insights paragraph routed through Arbiter's own router. No separate weekly email is sent.
